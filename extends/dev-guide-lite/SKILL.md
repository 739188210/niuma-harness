# dev-guide-lite · 轻量 TDD 开发指导

用于日常功能开发、Bug 修复、页面改动和小型重构。目标不是产出文档，而是让开发保持：**目标清楚、改动克制、TDD 优先、验证真实、页面改动必须浏览器验收**。

---

## 核心原则

1. **轻量优先**：不生成固定多阶段文档；默认只输出短计划和验证结果，不为流程完整写长篇。
2. **实用 TDD**：优先先写测试或复现步骤；Bug 修复优先先复现。项目不具备自动化测试条件时，用可执行验证步骤替代，但必须说明原因。
3. **最小实现**：只改与本次目标直接相关的代码。匹配项目现有风格，复用已有组件、工具、接口和模式；新增架构、抽象或依赖前说明必要性。修改公共代码时检查所有调用方影响；修改用户输入、接口、权限、数据、文件、外部调用时显式处理错误和边界。
4. **真实验证**：代码写完不算完成；适用于本次任务的测试、构建、类型检查、浏览器验证通过后，才算完成。不适用或无法运行的验证项必须说明原因和替代验证；不允许只用"代码看起来没问题"作为验证结论。
5. **页面改动必须尝试 Playwright**：只要影响用户可见的页面、组件、样式、交互或路由，就必须尝试用浏览器验证真实效果（细则见第 5 章）。因登录、环境、服务、账号等原因阻塞时，明确说明阻塞点并请求用户配合。

---

## 1. 适用范围

**适用于**：功能开发、Bug 修复、页面/组件/样式/交互改动、表单与路由调整、小型重构、单个接口或调用链调整、局部性能或体验优化。

**不适用于**：大型架构设计、跨系统重构、数据迁移、权限体系重建、支付/结算/资金链路重构、多服务协同改造，以及需要完整 PRD / 技术设计 / 实施计划的复杂需求。

遇到不适用场景，不要强行套用本 skill，先提醒用户需要更重的设计或单独确认方案。

---

## 2. 任务分级

按风险和影响范围分三类。

### quick

文案修改、样式微调、常量/枚举/配置小改、单文件小 Bug 等明确且低风险的改动。

```text
确认目标 → 修改 → 最小验证 → 汇报
```

### normal

普通功能开发、明确 Bug 修复、单个接口/页面/组件改动、表单/按钮/弹窗/交互调整、小型重构。

```text
目标 → 测试/复现 → 实现 → 自动测试 → Playwright（如适用）→ 汇报
```

### careful

权限、用户数据、支付、订单、导出、批量操作；公共组件/公共 API/公共工具修改；影响多个页面或调用方；代码量小但失败代价高；不可逆或难回滚的操作。

```text
简短方案 + 风险说明 → 必要时等用户确认 → 测试/复现 → 实现 → 验证 → Review → 汇报
```

`careful` 不等于重文档，但动手前必须说明风险。

---

## 3. 开发前最小对齐

开始前先给出简短计划：

```markdown
目标：...
级别：quick / normal / careful
验证方式：...
页面验证：需要 / 不需要，原因：...
计划：
1. ...
2. ...
```

需求不清楚时先问用户，不要猜。以下情况必须澄清：目标或成功标准不明确、页面入口或路径不明确、需要登录但缺少测试账号、涉及权限/数据/支付/批量等高风险行为、有多个合理实现方向且选择影响体验或架构。

---

## 4. 实用 TDD

默认采用实用 TDD，而非机械 TDD。

**Bug 修复**：①写一个能复现问题的自动化测试，成本过高则记录可执行复现步骤；②确认复现与用户描述一致；③写最小修复；④重跑测试或复现步骤，确认问题消失。

**新功能**：①写最小行为测试或验收用例；②实现刚好满足测试的功能；③补边界测试；④验证用户可见行为。

**页面改动**：可把 Playwright 操作步骤作为验收用例（打开页面 → 定位元素 → 操作 → 验证结果）。

**允许不用自动化测试的情况**（须说明原因并给出验证方式）：项目无测试框架、测试环境缺失、改动仅为文案或极小样式、自动化测试成本明显高于改动价值、当前任务更适合用 Playwright 验收。

### 质量检查

代码改完后按项目规则执行必要检查：单元测试、集成测试、类型检查、lint、build、Playwright 页面验证。检查失败时不要汇报完成——摘要失败原因、做最小修复、修复后重新验证；无法修复则说明阻塞点。

---

## 5. 页面改动 Playwright 规则

只要本次改动影响以下任一大类，必须尝试 Playwright：

- **结构与路由**：页面、路由、layout、导航
- **组件与交互**：组件、表单、按钮、弹窗、菜单、前端交互逻辑
- **样式与布局**：样式、响应式布局
- **状态展示**：加载态、错误态、空状态、权限展示

### Playwright 最小验证项

至少检查：①页面能打开、无白屏；②控制台无新增 `error`；③目标元素可见；④目标交互可完成；⑤页面结果符合预期；⑥涉及视觉/布局/样式时查看页面或截图；⑦涉及响应式时至少检查一个桌面宽度 + 一个移动宽度。

### 登录阻塞处理

页面需要登录时：①不伪造验证结果；②先判断是否已有登录态；③若无，提醒用户在浏览器手动登录；④需要账号时请求测试账号/测试环境账号，不要求生产账号或敏感密码；⑤用户登录后继续验证。阻塞时按第 8 章模板汇报。

### 环境阻塞处理

因环境问题无法验证时：明确失败命令、摘要关键错误、判断原因（依赖缺失 / 环境变量缺失 / 后端不可用 / 端口冲突 / 构建失败）；能最小修复则修复，不能则说明需要什么输入。

---

## 6. 服务启动

页面验证前必须确认服务状态。按优先级找启动方式：①用户提供的 URL；②已运行的本地服务；③项目脚本中的启动命令；④README / Makefile / 文档中的启动方式；⑤询问用户。

常见来源：`package.json` 的 scripts、README、Makefile、`docker-compose.yml`、项目本地脚本目录。**依赖锁文件（`pnpm-lock.yaml` / `yarn.lock` / `package-lock.json` / `bun.lockb`）只反映包管理器，不提供启动命令，不能作为来源。** 不要在不知道启动方式时乱试命令。

启动后：等待服务可访问 → 打开目标页面 → 执行 Playwright 验证 → 收集控制台错误和关键状态 → 汇报。若为本次验证临时启动服务，完成后必须汇报服务地址、端口、进程/后台任务状态，并询问用户是否保留或停止；不要静默遗留临时服务。启动失败按第 8 章模板汇报。

---

## 7. 高风险暂停点

遇到以下情况必须暂停并向用户确认，不能静默推进：

- 修改权限、鉴权、授权逻辑。
- 修改支付、订单、资金、结算相关逻辑。
- 修改用户数据、隐私数据、敏感字段。
- 修改数据库结构或迁移数据。
- 修改公共 API 契约。
- 修改公共组件并影响多个页面。
- 修改登录、Session、Token、Cookie、SSO。
- 修改批量操作、导入、导出、删除。
- 执行不可逆操作。
- 需要生产账号、生产数据或敏感凭据。
- 发现现有实现可能有安全问题或数据越权。

提问时按第 8 章模板给出风险、推荐方案、影响范围、回滚方式、需要确认的事项。

## 8. 汇报格式

所有汇报统一用以下模板，不写长文档。

### 完成汇报

```markdown
已完成：...
关键改动：...
验证结果：
- 测试：...
- 类型检查 / 构建：...
- Playwright：...
风险 / 未覆盖：...
```

未运行某项验证时说明：

```markdown
未运行：...
原因：...
替代验证：...
```

### 高风险暂停提问

```markdown
风险：...
推荐方案：...
影响范围：...
回滚方式：...
需要你确认：...
```

### 阻塞汇报（Playwright / 服务启动）

```markdown
状态：未完成
阻塞原因：需要登录 / 缺少测试账号 / 缺少环境变量 / 服务无法启动 / ...
已请求用户：...
待继续验证：...
登录或环境补齐后可继续：是 / 否
```

服务启动失败时替换为：

```markdown
服务启动：失败
命令：...
错误摘要：...
判断原因：...
下一步：已修复 / 需要用户提供 ...
```

## 9. 隔离工作区

当用户要求并行多任务、使用 worktree、避免任务互相污染时，为每个任务创建独立分支和独立工作目录。当前工作区已有未提交改动时，先提示用户并建议启用隔离工作区；只有用户确认后才创建 worktree。

### 触发条件

- 用户明确说"并行做多任务 / 用 worktree / 隔离开发"。
- 同一轮有多个相互独立的开发任务，或多 agent / 多 CLI 同时改代码。
- 当前工作区已有未提交改动，继续开发可能污染现场；此时只建议启用隔离工作区，需用户确认后再创建。

### 原则

1. 每个任务一个独立 Git 分支 + 一个独立 worktree 目录；一个会话只在一个 worktree 中工作。
2. worktree 只隔离工作目录，不保证最终合并无冲突。改共享文件、公共组件、schema、migration、lockfile 时额外提示冲突风险。
3. 启用前确认目标目录是 Git 仓库；不是则先问用户目标仓库路径。

### 工具适配

- **Claude Code**：优先用原生 worktree 能力。
- **Codex CLI / 其他工具**：用 `git worktree` 创建独立目录和分支，再把工具工作目录指向它。

### 默认不做

除非用户明确要求，不自动：push 到远程、创建 MR、删除 worktree、删除本地分支、丢弃未提交改动。

完成后按第 8 章汇报，并补充：隔离工作区是否启用、分支、worktree 路径、基础分支、后续是否需要 push / MR / 清理。

## 10. 禁止行为

以下行为在全文中只在此声明一次，适用于所有章节：

- 未确认高风险决策就修改（见第 7 章）。
- 不测试也不说明原因；用"应该没问题"代替验证。
- 为了通过测试而修改测试语义。
- 页面改动后不打开浏览器验证；只看代码不打开页面。
- 服务没启动却声称页面正常。
- 登录或环境受阻时跳过验证且不说明。
- 伪造测试、构建、浏览器验证结果。
- 测试或构建失败时汇报完成。
- 顺手重构、删除无关代码、引入无关依赖、改变未被要求改变的行为。
- 执行不可逆操作前不确认。
- 需要生产账号 / 敏感凭据时直接索取或使用。
