# AI 代码生成约束

## 1. 说明

本文档约束 AI 在项目中的代码生成行为,确保产出物符合项目规范、具备工程一致性、避免重复造轮子,并遵循实用 TDD。

---

## 2. 核心原则

### 2.1 优先复用,禁止重复造轮子

1. 生成代码前检索项目是否已有同类公共工具、常量、类型定义、错误处理、返回/响应格式、状态管理或数据访问封装
2. 使用项目已有实现,禁止创建功能重复的工具、模块、类型或封装
3. 不确定时通过文件搜索和内容搜索(如 `glob/grep`)确认,不得臆造
4. 确需新增公共能力时,说明理由并在实施清单中标注
5. 禁止以"更方便"为由绕过项目既有公共能力

### 2.2 整体架构意识,合理分层与封装

1. 明确文件、模块、类型、函数/方法的归属层次,禁止业务逻辑随意散落
2. 公共逻辑优先提取为独立函数/方法、组合能力或下沉到公共模块
3. 新增文件、模块、类型、函数/方法前评估是否已有合适位置,避免新建冗余文件
4. 修改既有代码时优先重构复用,而不是复制一份新的
5. 保持依赖方向正确,避免循环依赖

### 2.3 工程约定来源

代码生成前必须以当前项目文件校验 `PROJECT-CONTEXT.md` 中的技术栈与工程约定;事实源优先级遵循 `references/项目上下文指南.md` 第 3 节。

若 `PROJECT-CONTEXT.md` 缺失相关约定,或与当前项目扫描结果冲突,不得直接按 AI 常识生成代码;必须按 `references/项目上下文指南.md` 更新上下文或记录待确认项后再继续。

---

## 3. 实用 TDD 约束

本 skill 采用**实用 TDD**。业务逻辑默认测试先行;纯 boilerplate 可后补测试,但不得在没有任何验证的情况下声称完成。

### 3.1 测试用例来源

| 改动级别 | 测试用例来源 | 要求 |
|---|---|---|
| **微型** | `04-验证报告.md` 的最小验证项 | 记录验证命令、结果和关键观察点;不强制自动化测试 |
| **小型** | `03-实施清单.md` 的 RED 部分 | 必须写清楚输入、前置条件、预期输出 |
| **中型/大型** | `02-设计文档.md` 的测试策略章节 | 每条验收标准至少对应一个 `TC-XXX` 测试用例 |

### 3.2 通用执行规则

1. 业务逻辑必须遵循 `RED → GREEN → REFACTOR`:
   - RED:先写失败测试,并实际确认失败;
   - GREEN:写最小实现,并实际确认通过;
   - REFACTOR:清理实现,且测试仍通过。
2. 测试用例必须来自本节指定来源,不得在验证阶段临时补造测试来凑数。
3. 新增或修改业务代码后,必须运行相关测试;重构后必须重新运行测试。
4. 每个由本次改动引入的测试失败必须修复,除非用户明确确认延期。
5. 数据载体、类型定义、常量、配置、样式声明、纯样板代码(boilerplate)等可后补测试,但仍需通过编译、类型检查、构建或等价验证。

### 3.3 前端浏览器实际验证

涉及前端页面展示、组件交互、路由跳转、表单、弹窗、列表、状态反馈或视觉布局等用户可见行为时,不得只依赖单元测试、组件测试、类型检查、lint 或 build 结果声称完成。

必须通过真实浏览器实际打开相关页面验证。优先使用 Playwright MCP;若当前环境不可用,使用项目已有 E2E 工具(如 Playwright / Cypress);仍不可用时,使用本地浏览器或等价浏览器方式手动验证。

没有浏览器验证证据时,不得写"前端功能完成"、"页面正常"、"交互已验证"等结论。若当前环境无法启动页面或无法访问浏览器,必须在 `04-验证报告.md` 记录阻塞原因、是否影响验收,以及恢复后应执行的验证步骤。

---

## 4. 代码风格

1. 文件名、模块名、类型名、函数/方法名、变量名遵循项目既有命名风格
2. 分层、数据/状态访问、数据载体或类型定义命名遵循项目既有约定,以当前项目文件和经当前扫描校验后的 `PROJECT-CONTEXT.md` 为准,参考 `references/stack/<栈>.md`
3. 常量命名遵循项目既有约定;若项目无明确约定,再参考对应栈适配包
4. 保持与项目现有代码一致的缩进、换行、括号风格
5. 禁止无意义注释,必要注释说明"为什么"
6. 函数/方法职责单一,避免过长
7. 禁止魔法数字和硬编码字符串,优先使用常量、枚举、类型定义或配置

---

## 5. 入口与契约

1. 新增入口前确认项目已有路径、命名、参数传递、事件、命令或调用方式
2. 禁止随意修改已有接口、页面、组件、命令、任务、函数或模块的对外契约
3. 若项目存在统一响应格式、错误格式、组件事件契约、CLI 输出格式或函数返回约定,必须沿用
4. 对外契约必须考虑兼容性,破坏性变更需用户确认
5. 输入校验使用项目统一校验方式
6. 错误码、异常信息、错误对象或用户可见提示与项目规范保持一致

---

## 6. 数据、状态与持久化访问

1. 若涉及数据库、文件、缓存、远程 API、前端状态管理或本地持久化,必须使用项目已有访问方式
2. 禁止手写与项目既有数据访问、状态管理或持久化封装职责重复的实现
3. 若涉及事务、并发、缓存一致性、状态同步或重试补偿,必须明确边界和失败处理
4. 若涉及结构变更,如表结构、数据模型、类型定义、状态结构、配置 schema,必须遵循项目命名与迁移约定
5. 敏感字段按项目规范处理,禁止明文序列化、渲染、输出或记录日志
6. 复杂查询、数据转换或状态派生逻辑优先复用项目已有公共能力

---

## 7. 安全与合规

1. 禁止硬编码密钥、密码、Token、API Key
2. 用户输入必须校验,输出、渲染、查询和命令执行必须防注入、防 XSS 或防等价风险
3. 权限控制与项目现有安全框架或访问控制机制保持一致
4. 敏感操作按项目规范记录审计日志或等价追踪信息
5. 个人信息、凭证等数据按规范脱敏或加密

---

## 8. 测试

1. 新增业务代码必须同步生成或补充相关测试
2. 测试覆盖正常路径和主要异常路径
3. 测试命名、结构、工具与项目已有测试保持一致
4. 禁止生成无法运行的测试代码
5. 修改既有代码后评估是否需要同步修改测试
6. 测试数据优先使用项目已有 builder、factory、fixture、mock 或等价构造方式,避免魔法数字

---

## 9. 工程与提交

1. 代码生成后执行项目编译、类型检查、构建或等价校验命令
2. 执行项目代码检查/测试命令
3. 禁止提交未验证的代码
4. 文件创建位置符合项目目录结构、模块边界和职责划分
5. 删除代码前确认无其他引用
6. 代码修改小步快跑,避免一次性生成大量难以审查的代码

---

## 10. Agent 调用

编码前后,根据 `references/Agent调用与降级指南.md` 判断是否需要专家评审。

常见触发场景:

- 设计阶段:复杂架构问题调用架构评审,测试策略问题调用 TDD 指导。
- 编码完成后:中型及以上改动调用代码评审;涉及安全时调用安全评审。
- Agent 不可用时,按 `references/Agent调用与降级指南.md` 中对应角色的 checklist 自检,并将结论记录到当前阶段文档或验证报告。

---

## 11. 子代理执行

1. 子代理执行任务前,必须提供 `PROJECT-CONTEXT.md`、当前阶段文档、本约束文档,以及本次局部扫描结论或相关文件证据。
2. 子代理完成任务后,主 AI 必须验证其产出是否符合项目约束。
3. 多个子代理并行时,避免对同一文件产生冲突修改。

---

## 12. 红线清单

以下情况若未在已确认的 `00-基础资料.md`、`01-需求文档.md`、`02-设计文档.md` 或 `03-实施清单.md` 中授权,必须停止并询问用户;若已授权,记录依据并继续,只有超出已确认范围或风险升级时再询问。

微型点状改动例外:若已按 `SKILL.md` 判定为微型,且只是文案、枚举值、常量、参数、配置或明显 Bug 等点状修改,不改变业务规则、不改变对外契约、不涉及权限/金额/状态/数据一致性风险,可将用户原始请求、微型定级依据和 `04-验证报告.md` 中的最小验证项视为授权依据。

微型明显 Bug 还必须满足:仅恢复既定/期望行为,不重新定义业务规则或对外契约。一旦涉及公共契约变化、业务规则重定义、权限/金额/状态/数据一致性风险,必须升级流程或询问用户。

需要停止并询问用户的情况:

1. 需要引入新的外部依赖
2. 需要修改已有公共接口、公共类型、公共模块或对外契约
3. 需要新增模块或调整模块依赖关系
4. 发现项目已有实现与需求冲突
5. 通过代码扫描、引用搜索、LSP、现有文档或相似实现仍无法判断应采用哪种实现方式,且该选择会影响业务范围、对外契约、兼容性承诺或高风险技术取舍
6. 测试失败后用户未确认遗留,不能继续声称功能完成

以下行为禁止:

1. 生成与项目技术栈无关的代码
2. 生成无法编译、无法通过类型检查或明显存在语法错误的代码
3. 使用已弃用或项目未引入的库
4. 没有任何测试或等价验证的情况下声称功能完成
5. 擅自修改不属于当前需求范围的文件
6. 使用占位代码替代关键业务逻辑而不加标注

---

## 13. 一句话总结

AI 生成代码必须以项目现状为基线、以既有实现为参考、以公共复用为优先、以工程约定为准则、以测试先行为习惯,写出可编译、可测试、可维护、与项目风格一致的代码。
