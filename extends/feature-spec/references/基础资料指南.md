# 基础资料 · 编写指南

## 1. 说明

本文件记录本次需求的**原始输入**和**本次局部扫描快照**,是后续流程的起点。

- **用户提供原始输入**:通过自由描述讲清背景、问题、目标、约束和相关线索。
- **AI 整理为结构化文档**:把用户的口述/文字输入整理成 `00-基础资料.md`,统一格式,便于后续阶段引用。
- **AI 补充局部扫描快照**:根据本次需求范围扫描相关源码、配置、测试、迁移、入口和调用链,记录项目现状与证据。
- **缺失或不确定的信息**:先按第 4 节“查证 → 默认 → 询问”的顺序处理;只有会阻塞业务意图、验收标准或交付目标,且无法从项目证据确定时才追问用户。若暂时无法确认但不阻塞当前阶段,记录为“待确认项”,不能臆造或推测。

本文件定稿后,作为 `需求文档`、`设计文档` 或 `实施清单` 的输入。

---

## 2. 建议用户提供的内容

用户按需描述,能讲清就行。AI 整理时会按以下结构归入 `00-基础资料.md`。

### 2.1 背景

这次需求是从什么业务场景/问题来的?当前现状是什么?

### 2.2 问题

现在存在什么问题?影响范围多大?为什么现在要处理?

### 2.3 目标

希望达到什么总体结果?属于 新增 / 改造 / 兼容 / 收口 / 重构 中的哪一种?

### 2.4 已知约束

技术上、业务上、时间上有什么硬约束?

### 2.5 相关线索(尽量提供,能大幅减少 AI 臆测)

- 相关入口、接口、页面、命令、任务或模块路径
- 相关文件、类型、数据模型、状态结构、表结构、枚举或配置
- 可参考的已有功能
- 示例数据 / 输入输出样例

### 2.6 改动大小预判

用户可说明本次改动预期是微型/小型/中型/大型。AI 会据此提议流程。

## 3. 本次局部扫描快照

AI 整理 `00-基础资料.md` 时,除用户原始输入外,必须记录本次需求相关的局部扫描结论。

建议包含:

1. 涉及文件、入口、接口、页面、命令、任务或模块路径
2. 当前实现现状
3. 关键调用链、数据流或状态流
4. 相关测试、构建、配置、迁移或脚本
5. 与 `PROJECT-CONTEXT.md` 一致的事实
6. 与 `PROJECT-CONTEXT.md` 冲突或疑似过期的事实
7. 暂未确认但会影响后续设计/实施的问题

示例:

```markdown
## 本次局部扫描快照

- 相关入口:`src/...`
- 当前实现:`xxx` 已存在,但未覆盖 `yyy`
- 参考实现:`src/...`
- 与 `PROJECT-CONTEXT.md` 一致:测试框架仍为 `xxx`
- 与 `PROJECT-CONTEXT.md` 冲突:上下文记录构建命令为 `aaa`,当前 `package.json` 中实际为 `bbb`
- 待确认:是否允许调整已有接口返回结构
```

局部扫描快照只服务当前功能,不要直接替代 `PROJECT-CONTEXT.md`。后续只有长期稳定、跨功能有复用价值的事实,才在 `04-验证报告.md` 中列为待写回 Project Context 候选。

## 4. 需求清晰度检查

当本次流程会产出 `00-基础资料.md` 时，在进入 `需求文档` 或 `实施清单` 前，AI 必须判断当前业务意图是否已经足够清晰，能否写出可验收、可测试、可执行的后续文档。

需求清晰度检查的目标不是增加用户确认次数，而是防止 AI 在关键业务决策上臆测。低风险、可查证、可按项目惯例默认的事项应由 AI 自行处理并记录依据，不应频繁打断用户。

### 4.1 处理顺序

遇到不确定点时，按以下顺序处理：

1. **先查证**：能通过代码扫描、项目文档、`PROJECT-CONTEXT.md`、配置、测试、接口定义确认的问题，不问用户，由 AI 自行查证。
2. **再默认**：如果项目已有同类实现、明确约定或稳定上下文能支持一个默认方案，AI 可以采用该默认方案并记录依据。
3. **最后才询问用户**：只有当问题会影响业务范围、验收标准、交付目标、权限、金额、状态流转、数据一致性或对外契约，且无法从项目证据中确定时，才暂停并询问用户。

### 4.2 触发检查的情况

出现以下情况时，必须做清晰度检查，但不一定要问用户：

1. 用户描述存在模糊表达，例如“优化”“完善”“支持一下”“处理一下”“改造一下”“增强一下”。
2. 当前目标暂时无法写成清晰的“触发条件 + 预期结果”。
3. 范围内、范围外不清。
4. 存在多个合理业务方案。
5. 涉及权限、金额、数量、状态流转、数据一致性、对外契约、用户可见行为。
6. 用户描述与当前代码扫描结果、`PROJECT-CONTEXT.md` 或既有文档存在冲突。

### 4.3 必须询问用户的情况

只有以下情况必须询问用户：

1. 查证后仍无法确定业务范围、验收标准或交付目标。
2. 多个方案都合理，且项目没有明确默认方案或同类实现依据。
3. 用户描述与项目现状冲突，AI 无法判断以哪边为准。
4. 继续推进会引入破坏性变更、对外契约变化、权限/金额/状态/数据一致性风险。
5. 用户明确要求先确认再继续。

### 4.4 提问规则

需要问用户时：

1. 一次只问一个阻塞性问题。
2. 每个问题必须给出 AI 的推荐答案和依据。
3. 非阻塞问题不打断流程，记录到待确认项或后续设计阶段。
4. 用户回答后，立即更新 `00-基础资料.md` 的原始输入整理、待确认项或局部扫描快照。
5. 如果用户回答改变改动级别，必须重新定级。

### 4.5 完成条件

需求清晰度检查完成后，才能进入“业务意图确认”。

完成条件：

- [ ] 当前目标能用一到两句话准确描述。
- [ ] 范围内、范围外已经明确，或未明确部分不阻塞当前阶段。
- [ ] 不存在会影响业务范围、验收标准或交付目标的未闭合阻塞问题。
- [ ] 后续验收标准可以写成“触发条件 + 预期结果”。
- [ ] 改动级别已根据最新信息复核。

## 5. 进入下一阶段的条件

AI 整理完成 `00-基础资料.md` 后,必须先完成需求清晰度检查，再向用户呈现**意图摘要**(背景、问题、目标、约束、改动级别判断),请用户确认业务意图是否准确。

- **业务意图确认是硬性确认点**,所有信任模式下都必须执行;不受信任模式影响。
- 确认范围只限业务意图(背景/问题/目标/约束),**局部扫描快照不需用户确认**——这是 AI 扫描事实,由 AI 负责准确性。

**级别复核(初判为小型时强制):**

初判为小型的,必须基于本次局部扫描快照复核:确认确实不碰存量(不改已有公共类/接口/方法的逻辑·签名·行为)、不改 DB 结构(不加表/加字段/改字段类型/迁移)、不改对外契约(已有接口的 URL/入参/返回结构)。复核后若应为中型,升级并补走需求文档;微型和中大型不额外复核(微型代价小,中大型有需求/设计兜底)。

确认结果:

- 用户确认无误 → 按复核后的改动级别进入下一阶段(后续阶段的推进节奏按信任模式走)
- 用户提出修正 → 更新 `00-基础资料.md` 后重新呈现意图摘要,直到确认通过

意图确认通过后,按改动级别进入:

- 微型 → `验证报告`
- 小型 → `实施清单`
- 中型/大型 → `需求文档`

---

## 6. 一句话总结

用户提供原始输入，AI 补充本次局部扫描快照，并通过需求清晰度检查闭合会影响业务范围、验收标准或交付目标的阻塞问题，共同整理为结构化 `00-基础资料.md`;缺失信息先查证、再默认、最后才询问,技术实现类问题可记录到后续设计或实施阶段处理。
