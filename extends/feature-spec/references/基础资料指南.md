# 基础资料 · 编写指南

## 1. 说明

本文件记录本次需求的**原始输入**和**本次局部扫描快照**,是后续流程的起点。

- **用户提供原始输入**:通过自由描述讲清背景、问题、目标、约束和相关线索。
- **AI 整理为结构化文档**:把用户的口述/文字输入整理成 `00-基础资料.md`,统一格式,便于后续阶段引用。
- **AI 补充局部扫描快照**:根据本次需求范围扫描相关源码、配置、测试、迁移、入口和调用链,记录项目现状与证据。
- **缺失或不确定的信息**:AI 必须向用户追问;若暂时无法确认,记录为"待确认项",不能臆造或推测。

本文件定稿后,作为 `需求文档`、`设计文档` 或 `实施清单` 的输入。

---

## 2. 建议用户提供的内容

用户按需描述,能讲清就行。AI 整理时会按以下结构归入 `00-基础资料.md`。

### 2.1 背景

这次需求是从什么业务场景/问题来的?当前现状是什么?

### 2.2 问题

现在存在什么问题?影响范围多大?为什么现在要处理?

### 2.3 目标

希望达到什么总体结果?属于 新增 / 改造 / 兼容 / 收口 / 重构 中的哪一种?

### 2.4 已知约束

技术上、业务上、时间上有什么硬约束?

### 2.5 相关线索(尽量提供,能大幅减少 AI 臆测)

- 相关入口、接口、页面、命令、任务或模块路径
- 相关文件、类型、数据模型、状态结构、表结构、枚举或配置
- 可参考的已有功能
- 示例数据 / 输入输出样例

### 2.6 改动大小预判

用户可说明本次改动预期是微型/小型/中型/大型。AI 会据此提议流程。

## 3. 本次局部扫描快照

AI 整理 `00-基础资料.md` 时,除用户原始输入外,必须记录本次需求相关的局部扫描结论。

建议包含:

1. 涉及文件、入口、接口、页面、命令、任务或模块路径
2. 当前实现现状
3. 关键调用链、数据流或状态流
4. 相关测试、构建、配置、迁移或脚本
5. 与 `PROJECT-CONTEXT.md` 一致的事实
6. 与 `PROJECT-CONTEXT.md` 冲突或疑似过期的事实
7. 暂未确认但会影响后续设计/实施的问题

示例:

```markdown
## 本次局部扫描快照

- 相关入口:`src/...`
- 当前实现:`xxx` 已存在,但未覆盖 `yyy`
- 参考实现:`src/...`
- 与 `PROJECT-CONTEXT.md` 一致:测试框架仍为 `xxx`
- 与 `PROJECT-CONTEXT.md` 冲突:上下文记录构建命令为 `aaa`,当前 `package.json` 中实际为 `bbb`
- 待确认:是否允许调整已有接口返回结构
```

局部扫描快照只服务当前功能,不要直接替代 `PROJECT-CONTEXT.md`。后续只有长期稳定、跨功能有复用价值的事实,才在 `04-验证报告.md` 中列为待写回 Project Context 候选。

## 4. 进入下一阶段的条件

AI 整理完成 `00-基础资料.md` 后,必须先向用户呈现**意图摘要**(背景、问题、目标、约束、改动级别判断),请用户确认业务意图是否准确。

- **业务意图确认是硬性确认点**,所有信任模式下都必须执行;不受信任模式影响。
- 确认范围只限业务意图(背景/问题/目标/约束),**局部扫描快照不需用户确认**——这是 AI 扫描事实,由 AI 负责准确性。

**级别复核(初判为小型时强制):**

初判为小型的,必须基于本次局部扫描快照复核:确认确实不碰存量(不改已有公共类/接口/方法的逻辑·签名·行为)、不改 DB 结构(不加表/加字段/改字段类型/迁移)、不改对外契约(已有接口的 URL/入参/返回结构)。复核后若应为中型,升级并补走需求文档;微型和中大型不额外复核(微型代价小,中大型有需求/设计兜底)。

确认结果:

- 用户确认无误 → 按复核后的改动级别进入下一阶段(后续阶段的推进节奏按信任模式走)
- 用户提出修正 → 更新 `00-基础资料.md` 后重新呈现意图摘要,直到确认通过

意图确认通过后,按改动级别进入:

- 微型 → `验证报告`
- 小型 → `实施清单`
- 中型/大型 → `需求文档`

---

## 5. 一句话总结

用户提供原始输入,AI 补充本次局部扫描快照,共同整理为结构化 `00-基础资料.md`;缺失信息必须追问,不确定项记录为待确认。
