# 实施清单

## 1. 说明

本文档用于将当前阶段已确认的内容拆解为可直接执行的落地动作。

- **小型改动**:基于 `00-基础资料.md`、经当前扫描校验后的 `PROJECT-CONTEXT.md` 和必要的实施前复扫结果生成。
- **中型/大型改动**:基于已定稿的 `02-设计文档.md` 生成;若实施前发现事实冲突,先按 `references/项目上下文指南.md` 第 3 节判定事实源优先级,再按 `references/项目上下文指南.md` 第 7 节执行增量更新或完整扫描。

本文档的目标不是再次讨论需求或设计,而是将已确认内容拆解为可以直接执行的落地动作,每个动作按 **TDD 循环(RED→GREEN→REFACTOR)** 组织。

除非需求明确要求分批或骨架先行,实施清单默认应以本批可验证、可运行、可交付为目标;不得把关键业务逻辑无故推迟为后续批次。

本文档定稿后,可作为开发实施、AI 执行或阶段验收的直接依据。

---

## 2. 本批实施目标

说明本批次实施要完成的结果。

建议聚焦以下内容:

1. 本批要打通什么链路
2. 本批要搭建什么骨架
3. 本批要补齐什么能力
4. 本批完成后的预期状态是什么

若存在分批推进,当前文档只描述本批,不混入后续批次细节。

## 4. 分工边界

明确本批实施负责什么,不负责什么。

#### 4.1 本批负责

明确本批应完成的动作。

#### 4.2 本批不负责

明确本批不处理的内容,例如:

1. 真实 SQL 调优
2. 联调
3. 数据准备
4. 线上发布
5. 字段翻译
6. 性能优化

此部分越清楚,越能避免范围失控。

## 5. 实施原则

说明实施阶段必须遵守的约束。

建议至少包括:

1. 是否允许改现有接口签名
2. 是否允许破坏既有逻辑
3. 是否优先复用已有代码模式
4. 是否允许仅做占位实现
5. 是否要求本批代码可编译
6. 是否要求本批能力可直接可用
7. 是否只记录稳定边界、路径模式和关键对象类型,避免维护易变的文件/函数枚举

## 6. TDD 垂直切片实施清单

默认按**可验证行为切片**拆解落地动作,而不是按文件、分层或技术对象横向拆分。每个切片应对应一个可观察行为、一个 `REQ-XXX` 或一个 `TC-XXX`,并按 **RED→GREEN→REFACTOR** 循环组织。

测试用例来源遵循 `references/AI代码生成约束.md` 的 TDD 规则:

- 小型改动:本节 RED 部分就是测试用例来源,必须写清楚输入、前置条件、预期结果。
- 中型/大型改动:本节 RED 部分引用 `02-设计文档.md` 测试策略中的 `TC-XXX` 编号,不重新定义测试用例。

实施清单应承接设计文档中的模块边界、seam、复用点和测试入口,把它们转成当前切片的执行约束。若实施中确认了新的项目级稳定事实,当前切片只能标注为待写回候选提示,它不是写回队列;验证阶段必须统一筛选并登记到 `04-验证报告.md` 第 12 节,否则视为未登记。

### 6.1 禁止横向切片

实施清单不得默认按“先类型、再数据层、再业务层、再入口层、最后测试”的方式横向推进。

默认必须按“行为切片”组织,每个切片都应能完成一个可观察行为:

- **RED**:写一个失败测试,覆盖一个 `REQ-XXX` 或一个 `TC-XXX`。
- **GREEN**:写最小实现,让该测试通过。
- **REFACTOR**:清理当前切片,不改变外部行为。
- **VERIFY**:运行当前切片相关测试或验证步骤。

只有当某些基础定义是多个切片的必要前置条件时,才允许先做 prefactoring,并且必须说明:

1. 为什么它是前置;
2. 如何验证它没有改变外部行为;
3. 它服务哪些后续切片。

## 7. 建议的落地顺序

说明本批实施的推荐执行顺序。优先按切片依赖关系排序,而不是按技术层排序。

推荐顺序:

1. 先做最小 tracer bullet:打通一条最窄、最关键、可验证的端到端路径。
2. 再做主要正常路径。
3. 再做异常路径、边界条件和兼容性。
4. 最后做清理、回归和非功能性验证。

如果必须先做 prefactoring,需要说明它服务哪些行为切片,以及如何验证它没有改变外部行为。

同时建议说明排序原因,便于不同执行者保持一致。

## 8. 本批明确不做的事项

用显式列表列出本批故意不处理的内容。

这部分建议写得具体,不要只写"其他事项"。

示例方向:

1. 不做真实业务逻辑细节
2. 不做真实查询 SQL 调优
3. 不做联调验证
4. 不做历史逻辑收口
5. 不做性能优化
6. 不做字段清洗或翻译

## 9. 验收标准

> 小型改动不产出 `01-需求文档.md` 时，本节验收标准即作为 `04-验证报告.md` 的验收基线；必须在本节直接定义`REQ-001`、`REQ-002` 等编号，供验证报告引用。
>
> 中型/大型改动的 `REQ-XXX` 来源仍为`01-需求文档.md`，本节只引用和承接，不重新编号。

建议拆成两个部分。

### 9.1 必须满足

列出本批结束时必须满足的结构性、行为性或编译性要求。

### 9.2 暂不要求

列出本批不纳入验收范围的内容。

此部分用于防止按"最终可用"错误验收"阶段性骨架"。

## 11. Agent 调用建议

实施清单完成后,进入编码前,评估是否调用 **TDD 指导**角色:

- 测试复杂 / Mock 边界模糊 / 测试框架不熟悉。

编码完成后评估:

- 中型及以上 / 修改既有代码 → 代码评审;
- 涉及用户输入、认证、数据库、外部 API、敏感数据 → 安全评审。

调用输入、输出要求和降级自检规则见 `references/Agent调用与降级指南.md`。

## 12. 给其他 AI 的执行提示

若后续由其他 AI 或其他执行者继续处理,可在此处补充统一约束。

建议包括:

1. 本批目标是什么
2. 禁止越界到哪些后续内容
3. 以什么为优先原则
4. 哪些地方只能做占位
5. 哪些地方必须保证不破坏现有行为

若没有交接需求,可写"当前无"。

## 13. 一句话总结

用一句话总结本批实施的本质。

建议强调:

1. 本批是在补骨架、补能力还是补细节
2. 本批交付的重点是结构到位还是功能可用
3. 后续下一批将在哪个基础上继续推进
