# 验证报告 · 编写指南

## 1. 说明

本文档是功能"正确性、稳定性"的闭环凭证,采用**活文档模式**:在执行过程中逐步填充,不是最后一次性编写。

验证报告的核心不是复述测试输出,而是回答三个问题:

1. 本次需求是否达到全部验收标准?
2. 有什么遗留风险和下一步?
3. 是否可进入下一批 / 上线?

验证报告只记录测试与验证执行结果,不重新定义验收标准或测试用例。验收标准与测试来源遵循 `references/AI代码生成约束.md` 和对应阶段文档:

- 微型改动:不强制定义 `REQ-XXX`,记录最小验证项;
- 小型改动:验收标准来源于 `03-实施清单.md` 第 9 节的 `REQ-XXX`,测试来源于 `03-实施清单.md` 第 6 节的 RED 项;
- 中型/大型改动:验收标准来源于 `01-需求文档.md` 第 10 节的 `REQ-XXX`,测试来源于 `02-设计文档.md` 第 7 节测试策略中的 `TC-XXX` 用例。

---

## 2. 验证范围

说明本次验证覆盖的功能点、接口、模块;同时显式列出**不覆盖**的部分及原因。

### 微型改动的验证

微型改动(改文案、加枚举、调参数、修明显 bug)不强制要求写自动化测试,但必须在验证报告中记录:

1. 验证命令(如编译命令、手动验证步骤)
2. 验证结果(通过/失败)
3. 关键观察点

示例:

```markdown
## 验证记录

| 验证项 | 验证命令/步骤 | 结果 | 备注 |
|---|---|---|---|
| 枚举值新增后编译通过 | 项目校验命令(见 `references/stack/<栈>.md`) | ✅ 通过 | |
| 错误提示文案已更新 | 手动调用接口,观察返回文案 | ✅ 通过 | 返回"用户不存在" |
```

## 3. 验收结论

核心表格。将**既定验收标准**映射到实际执行的测试用例或验证步骤及结果。

| 需求编号 | 验收标准 | 测试文件/测试单元/浏览器工具 | 测试用例/验证步骤 | 结果 | 备注 |
|---|---|---|---|---|---|
| REQ-001 | 给定输入 A,当执行操作 B 时,得到结果 C | <测试文件、测试单元或浏览器工具> | <测试用例名或验证步骤> | ✅ 通过 | |
| REQ-002 | 给定异常条件 X,当执行操作 Y 时,得到错误结果 Z | <测试文件、测试单元或浏览器工具> | <测试用例名或验证步骤> | ❌ 失败 | 详见遗留问题 |

前端改动的浏览器实际验证需记录使用工具(Playwright MCP / Playwright / Cypress / 手动浏览器)、页面 URL 或路由、操作步骤、实际观察结果,以及截图、trace、录像、控制台错误或网络错误等证据路径(如有)。若未执行浏览器验证,必须记录原因、是否阻塞验收,以及恢复后应执行的验证步骤。

## 9. 测试失败处理规则

验收结论表格中出现 ❌ 失败时,AI 必须按以下规则处理,不能简单把失败塞到"遗留问题"里糊弄过去。

### 9.1 本次改动引入的失败必须修复

- 因本次改动导致的测试失败,必须修复后才能继续。
- 不得将"我还没修好"直接记录为遗留问题。

### 9.2 允许记录为遗留的严格条件

只有满足以下条件之一,才允许把失败记录到"遗留风险与下一步":

1. **用户明确确认延期**:用户在对话中明确说"这个先遗留,下一批处理"。
2. **改动前已存在问题**:失败在改动前就已经存在,且能提供改动前的验证证据(例如改动前运行同一测试也失败、或引用历史 issue/提交)。
3. **外部环境缺失导致无法执行**:例如依赖的外部服务、数据库、中间件在当前环境不可用,且已清楚说明缺失条件和恢复后的验证步骤。

### 9.3 对结论的强制约束

- 任何失败未处理且未满足上述遗留条件时,结论中**不得**写"功能完成""可上线""验收通过"等字样。
- 结论必须诚实反映当前状态,例如:"除 REQ-003 外,其余验收标准已满足;REQ-003 因 XX 原因待用户确认是否遗留"。

### 9.4 用户未回复时的处理

若发现失败需要用户决策(是否遗留、是否调整需求),AI 必须暂停并提问,不能自行假设用户同意遗留。

## 11. 活文档填充规则

执行过程中,每完成一个 TDD 循环,立即更新本章:

1. 把测试方法及结果填入"验收结论"表格
2. 如有失败,记录到"关键执行摘要"
3. 如发现新的遗留问题,记录到"遗留风险与下一步"
4. 实施中的小调整,记录到"实施中调整项"

## 12. 待写回 Project Context

执行过程中发现的、值得沉淀到 `PROJECT-CONTEXT.md` 的稳定事实,先记录在此,待功能全部完成后统一筛选并写回。

| 事实 | 类型 | 证据 | 是否稳定 | 建议动作 | 是否已写回 |
|---|---|---|---|---|---|
| 示例:项目新增 `common-util` 模块 | 模块结构 | `path/to/file` | 是 | 写回 | 否 |
| 示例:统一返回体增加 `traceId` 字段 | 非功能性约束 | `path/to/file` | 是 | 写回 | 否 |
| 示例:某外部服务当前测试环境不可用 | 临时状态 | 验证命令输出 | 否 | 不写回 | 否 |
| 示例:旧 README 与当前配置不一致 | 冲突事实 | `README.md` / `config.xxx` | 待确认 | 暂缓 | 否 |

筛选规则:

1. 只有长期稳定、跨功能有复用价值、且有当前项目文件证据或用户确认支撑的事实才允许写回 `PROJECT-CONTEXT.md`。
2. 临时状态、一次性问题、调试过程、未确认推测不得写回。
3. 与 `PROJECT-CONTEXT.md` 冲突的事实,必须标注证据和稳定性;高风险冲突需用户确认后再写回。

写回时机:

功能全部完成后,统一整理并写回 `PROJECT-CONTEXT.md`。禁止实时写回,避免中途频繁修改活文档。

## 13. 流程环节回顾

本次功能完成后,回顾各环节和 Agent 调用是否对决策产生了实际影响。这些数据用于持续优化流程,判断哪些环节有效、哪些可以精简。

判断标准:该环节/调用的产出是否改变了后续任何决策。若只是确认了已知结论,填"否"。

### 13.1 流程环节

| 环节 | 是否改变决策 | 说明 |
|---|---|---|
| 分级判定 | 是/否 | 如:复核后从小型升级为中型 |
| 意图确认 | 是/否 | 如:用户纠正了目标理解 |
| 局部扫描快照 | 是/否 | 如:发现了用户未提及的XX工具类 |
| TDD 循环 | 是/否 | 如:RED阶段暴露了边界问题 |

### 13.2 Agent 调用

| 阶段 | 评审角色 | 是否触发 | 采用方式 | 是否改变决策 | 说明 |
|---|---|---|---|---|---|
| 需求 | 计划评审 | 是/否 | Agent调用/降级自检/跳过 | 是/否 | 如:自检发现范围遗漏 |
| 设计 | 架构评审 | 是/否 | 同上 | 是/否 | |
| 设计 | TDD指导 | 是/否 | 同上 | 是/否 | |
| 编码后 | 代码评审 | 是/否 | 同上 | 是/否 | |
| 编码后 | 安全评审 | 是/否 | 同上 | 是/否 | |

### 13.3 回退记录

| 回退时机 | 从哪→到哪 | 原因 | 是否影响交付 |
|---|---|---|---|
| 如:实施阶段 | 实施→需求 | 发现验收标准遗漏异步场景 | 是,补了 REQ-003 |

无回退则填"无"。

- 已闭环:Y 项
- 未闭环:Z 项(列出并说明影响)

## 14. 一句话总结

验收标准必须有测试兜底,验证报告用"验收标准 → 测试 → 结果"证明每一条都成立。
