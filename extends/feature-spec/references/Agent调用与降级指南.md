# Agent 调用与降级指南

## 1. 说明

本 skill 将专业能力抽象为**角色**，即**AGENT**，推荐在关键节点调用专业 Agent 做评审,以提升需求、设计、测试、代码和安全质量。

必须采用**推荐调用 + 降级自检**策略:

- **优先**:与当前技术栈匹配的专用 Agent 做专业评审;
- **其次:** 通用Agent,例如 `code-reviewer`、`architect`、`planner`。
- **降级**:若 Agent 不可用,由主 AI 按本文对应角色的 checklist 自检,并把自检结论记录到当前阶段文档或验证报告中。
- **禁止**：禁止调用与当前技术栈明显不匹配的 Agent

> 不能把"没有 Agent"当成跳过评审的借口;也不能假装调用了不存在的 Agent。

---

## 2. 角色、触发场景与调用强度

Agent 名称以当前运行环境实际可用的 Agent 列表为准, 若当前环境没有对应 Agent，则按对应角色的 checklist 自检。

| 角色 | Agent | 建议调用场景 | 小型 | 中型 | 大型 |
|---|---|---|---|---|---|
| 计划评审 | `planner` | 需求复杂、跨模块、需拆解实施顺序 | 不调 | 需求复杂时调 | 建议调 |
| 架构评审 | `architect` | 新增模块、改公共接口/类、数据库结构变更、多方案取舍 | 不调 | 建议调 | 必须调 |
| TDD 指导 | `tdd-guide` | 测试复杂、Mock 边界模糊、测试框架不熟悉 | 可选 | 建议调 | 必须调 |
| 代码评审 | `code-reviewer` | 中型及以上、修改既有代码、涉及并发/事务/异常 | 建议调 | 建议调 | 必须调 |
| 安全评审 | `security-reviewer` | 涉及用户输入、认证授权、数据库、文件、外部 API、敏感数据 | 涉及安全时调 | 涉及安全时调 | 必须调 |

---

## 3. 调用输入规范

调用 Agent 时,必须提供以下上下文:

1. **当前阶段文档**:需求文档/设计文档/实施清单等
2. **`PROJECT-CONTEXT.md`**
3. **本次局部扫描结论**:`00-基础资料.md` 中的局部扫描快照、相关文件路径或证据片段
4. **明确的问题清单**:请 Agent 评审什么、输出什么

涉及设计、实施或评审时,必须提供足以校验 `PROJECT-CONTEXT.md` 的本次证据,避免 Agent 只依赖可能过期的全局上下文。

示例:

```markdown
当前阶段:设计文档评审
背景:
- 需求:新增某个业务能力或用户操作流程
- 影响范围:<模块/页面/接口/命令/任务>
- 本次扫描:<相关文件、入口、调用链、配置、测试>
- 上下文偏差:<如当前项目文件与 `PROJECT-CONTEXT.md` 是否存在冲突>
- 已有模式:参考 <项目内相似实现>
待评审问题:
1. 当前落点是否符合项目既有分层/模块边界?
2. 状态变化、幂等性、错误处理或回滚策略是否清楚?
3. 是否应复用现有错误处理、返回格式、事件机制或输出契约?
请输出:评审意见 + 风险等级(高/中/低) + 是否建议调整设计
```

---

## 4. 调用输出规范

Agent 或降级自检必须输出结构化结论,至少包含:

1. **结论**:通过 / 需调整 / 阻塞
2. **风险等级**:高 / 中 / 低
3. **主要发现**:列出具体问题,说明影响范围
4. **建议动作**:说明是否需要修改需求、设计、实施清单或代码
5. **是否阻塞下一阶段**:明确可以继续、需要调整后继续,还是必须暂停问用户

若无问题,也必须显式写明"未发现阻塞问题",不能省略结论。

## 5. Agent 不存在时的降级自检

### 5.1 planner 自检清单

- [ ] 需求是否拆成了可独立实施的子任务?
- [ ] 是否识别了任务间的依赖关系?
- [ ] 是否给出了合理的实施顺序?
- [ ] 是否有范围外明确排除的内容?

### 5.2 architect 自检清单

- [ ] 新增模块/类是否有清晰职责?
- [ ] 是否复用了现有公共能力?
- [ ] 是否避免了循环依赖?
- [ ] 公共接口/数据契约是否发生变化?
- [ ] 数据库变更是否影响现有数据?
- [ ] 扩展点是否预留?

### 5.3 tdd-guide 自检清单

- [ ] 每个验收标准是否至少有一个测试用例?
- [ ] 测试分层是否清晰(单元/集成/契约)?
- [ ] 外部依赖是否有 Mock 方案?
- [ ] 测试数据是否可构造?
- [ ] 边界条件和异常路径是否覆盖?

### 5.4 code-reviewer 自检清单

- [ ] 代码是否遵循项目命名和分层约定?
- [ ] 函数/方法是否职责单一、长度合理?
- [ ] 是否有重复代码可提取?
- [ ] 是否有无意义注释或调试代码?
- [ ] 错误处理是否显式?
- [ ] 是否有硬编码或魔法数字?

### 5.5 security-reviewer 自检清单

- [ ] 用户输入是否都做了校验?
- [ ] 数据库操作是否有 SQL 注入风险?
- [ ] 输出是否做了防 XSS 处理?
- [ ] 敏感数据是否脱敏/加密?
- [ ] 权限控制是否符合项目规范?
- [ ] 是否有审计日志要求?

---

## 6. 自检结果处理

- 自检发现的问题,必须记录到当前阶段文档的"待确认问题"或"风险"章节
- 高风险问题必须停下来问用户
- 中低风险问题可在实施中修复,并记录到验证报告

---

## 7. 一句话总结

Agent 能调则调,调不了就按 checklist 自检,不能把"没有 Agent"当成跳过评审的借口。
