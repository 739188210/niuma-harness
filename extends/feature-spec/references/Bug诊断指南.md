# Bug 诊断指南

## 1. 目标

Bug 修复的第一目标不是立刻改代码,而是建立能证明问题存在、也能证明问题被修复的反馈环。

本指南适用于“不明显 Bug”:无法稳定复现、根因不明、涉及状态/并发/缓存/数据库/外部 API、或修复可能改变既有业务行为的问题。明显 Bug 仍可按改动大小分级走轻量流程。

---

## 2. 诊断流程

### 2.1 建立反馈环

优先尝试以下反馈环,按能最快稳定复现用户症状的方式选择:

1. 失败测试:单元、集成、契约或 E2E 测试。
2. `curl` / HTTP 脚本:针对接口或服务端入口。
3. CLI 命令:用固定输入驱动命令、任务或脚本。
4. 浏览器脚本 / Playwright:驱动页面、断言 DOM、控制台或网络结果。
5. 捕获真实请求、事件或日志后重放。
6. 最小 throwaway harness:只启动必要模块或直接调用目标函数。
7. 人工步骤:最后手段,必须写成可重复执行的步骤,并记录观察点。

反馈环完成标准:

- [ ] 已实际运行过,并记录命令、步骤或脚本路径。
- [ ] 能触发用户描述的具体症状,不是附近的其他错误。
- [ ] 结果可判断红/绿,能在修复前失败、修复后通过。
- [ ] 尽量快、稳定、可重复;若是偶现问题,应提高复现概率并记录复现率。

没有可用反馈环时,不得直接猜测修复。应记录已尝试的复现方式,并向用户索要环境、日志、HAR、请求样例、录屏或允许添加临时诊断信息。

---

### 2.2 复现并最小化

先确认反馈环触发的是用户描述的同一个问题。

确认内容:

1. 实际症状与用户描述一致。
2. 错误信息、错误输出、错误页面、错误数据或性能表现已记录。
3. 多次运行结果稳定;偶现问题需记录复现次数与复现率。

复现后逐步最小化:

- 减少输入字段;
- 减少操作步骤;
- 减少依赖服务;
- 减少测试数据;
- 缩小调用入口;
- 去掉与症状无关的配置。

每次只移除一个因素,并重新运行反馈环。目标是尽量最小化到足以稳定触发同一症状的输入、步骤或环境集合。若因外部环境不可控、偶现概率过低、成本过高等原因无法继续最小化,必须记录原因。完成标准是证据足以支持根因判断和回归验证,而不是必须证明每个因素都是必要条件。

---

### 2.3 提出可证伪假设

在改代码前,提出按可能性排序的假设。根因不明时提出 3-5 个假设;若反馈环和最小化步骤已经把根因收敛到唯一可证据支持的原因,可只记录 1 个主假设和排除依据。

每个假设必须写成可证伪形式:

```text
如果原因是 X,那么观察/改变 Y 会出现 Z。
```

要求:

1. 不得只有一个无证据猜测就直接修;若只记录 1 个主假设,必须说明反馈环和最小化证据为何足以支撑它。
2. 每个假设必须能通过测试、日志、断点、对比实验或配置变化验证。
3. 假设之间尽量覆盖不同原因类别,例如输入、状态、权限、缓存、并发、数据、外部依赖、渲染。
4. 若用户在线,可向用户展示假设排序;用户不在时按当前证据继续。

---

### 2.4 验证假设

一次只验证一个变量。

优先使用:

1. 测试或最小实验;
2. 调试器 / REPL / 断点;
3. 局部、带唯一前缀的临时日志;
4. 配置或输入的最小变更。

禁止:

- 大面积打日志;
- 同时改多个变量;
- 顺手重构无关代码;
- 在根因未确认前扩大修改范围;
- 把临时诊断代码留在最终实现里。

临时日志必须使用唯一前缀,例如 `[DEBUG-xxxx]`,修复完成前必须搜索并清理。

---

### 2.5 修复和回归测试

确认根因后再修复。

若存在合适 seam,必须先把最小复现转成失败测试,再写修复代码。合适 seam 指能覆盖真实 bug 触发路径的测试入口,而不是只能测到内部细节的浅层入口。

修复要求:

1. 先让回归测试从失败变通过。
2. 再运行原始反馈环,确认用户症状消失。
3. 运行相关测试、编译、类型检查或构建命令。
4. 清理 `[DEBUG-...]` 日志、临时脚本和 throwaway harness;若需要保留,必须移动到明确标记的调试位置并说明原因。
5. 若没有合适 seam 写回归测试,必须在验证报告中说明原因和风险。

---

### 2.6 记录到验证报告

`04-验证报告.md` 必须记录以下内容:

1. 用户描述的症状;
2. 复现方式:命令、测试、脚本、浏览器步骤或人工步骤;
3. 最小复现:最小输入、最小数据、最小步骤;
4. 根因:最终确认的原因和证据;
5. 修复方式:改了什么,为什么能消除根因;
6. 回归测试:新增或修改的测试、运行命令和结果;
7. 原始反馈环复测结果;
8. 未覆盖风险:没有测试 seam、外部环境缺失、偶现未完全消除等。

若仍有失败或无法验证的项,必须遵守 `references/验证报告指南.md` 的测试失败处理规则,不得写“功能完成”“验收通过”或“可上线”。

---

## 3. 一句话总结

不明显 Bug 先建立红/绿反馈环,再最小化、假设、验证、修复和回归;没有能证明问题被修复的反馈环,就不能声称 Bug 已修好。
