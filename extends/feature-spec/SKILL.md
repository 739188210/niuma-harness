# feature-spec · 功能开发全流程引导

> 本 skill **语言/技术栈无关**:流程骨架(阶段、断点、TDD、信任模式、回退、Agent)通用。各语言/栈的分层、命名、构建、测试等**具体约定**由 `PROJECT-CONTEXT.md` 承载,并参考对应**栈适配包**(`references/stack/<栈>.md`,如 `java.md`、`vue-ts.md`、`python.md`)。

本 skill 把一次功能开发拆成固定阶段,但阶段数量根据**改动大小**动态调整。核心机制依然是**已产出的文档就是流程状态**,支持断点续传。

---

## 始终遵守的约束

1. 阶段开始前,先确认项目根目录的 `PROJECT-CONTEXT.md` 是否存在、内容有效且未过期。默认路径为项目根目录,文件名为 `PROJECT-CONTEXT.md`。

   若不存在、内容为空/仅含占位符、已过期,或项目大改,先做/更新 `PROJECT-CONTEXT.md`。过期判断标准见 `references/项目上下文指南.md` 第 4 节。
2. 做设计、写代码前,**必须先读 `PROJECT-CONTEXT.md`** 的稳定部分,易变信息通过文件搜索、内容搜索、符号引用(如 `glob/grep/LSP`)动态查询。

   `PROJECT-CONTEXT.md` 是导航索引和历史基线。事实源优先级遵循 `references/项目上下文指南.md` 第 3 节。
3. 进入“设计”“实施”“验证”阶段前,读 `references/AI代码生成约束.md` 并严格遵守;同时按本次触及的栈读对应 `references/stack/<栈>.md` 适配包(见「栈适配包」)。
4. 默认按**普通模式**推进,高风险必须停下来问用户,低风险自动推进。
5. 除 `trust` 模式按信任模式表最后统一汇总外,每个阶段产出后向用户汇报关键结论、产物路径和风险;无高风险/阻塞项则自动继续下一阶段,存在高风险/阻塞项时必须暂停并提问;用户明确反对时按`回退机制`处理。
6. 执行过程中发现新的稳定项目事实,先记录到 `04-验证报告.md` 的"待写回 Project Context"小节;写回筛选规则见 `references/验证报告指南.md` 第 12 节。功能全部完成后统一整理并写回 `PROJECT-CONTEXT.md`,禁止实时写回。

## 栈适配包

阶段①必须按 `references/项目上下文指南.md` 识别项目主栈或多栈范围,并写入 `PROJECT-CONTEXT.md`。

进入设计、实施、验证阶段前(中型/大型才有设计阶段),按本次改动**触及的栈**读取对应 `references/stack/<栈>.md` 适配包;多栈则加载多个。

若 `PROJECT-CONTEXT.md` 缺少技术栈信息,或当前扫描与上下文冲突,必须以当前项目文件为准,回到阶段①更新 `PROJECT-CONTEXT.md` 或记录待确认项。若某栈暂无适配包,依靠当前项目扫描 + 经校验的 `PROJECT-CONTEXT.md` 作为主要依据推进。

## 改动大小分级

分级是流程入口,决定后续路径、产物和 Agent 调用,**必须按依据判定,不凭直觉**。

判定采用**有序排除法**:从高到低依次回答"是/否",命中即锁定该级别、不再往下看;都不命中则为微型。

**1. 大型(满足任一)**

- 架构级改造 / 跨模块重构 / 数据迁移 / 不可逆破坏性变更

**2. 中型(满足任一)**

- 新增**完整功能模块**(新实体/新能力,含多条调用链)
- 改 **DB 结构**(加表/加字段/改字段类型/迁移)
- **碰存量**:修改已有公共类/接口/方法的逻辑·签名·行为
- 引入新依赖
- 改对外契约(已有接口的 URL/入参/返回结构)

**3. 小型**

- 新增或改变**单条**端到端调用链(单一接口/方法)

**4. 微型(以上都不是)**

- 仅点状修改:文案/枚举值/常量/参数/配置/修明显 bug;不新开链路,且除“让明显 Bug 回到既定/期望行为”外,不改变既有业务规则或对外契约

**裁决规则:**

1. **向上取整**:命中多级信号取最高级;判定有疑问取更高一级。
2. **增量豁免**:纯新增(加常量/加枚举值/加新方法)不算"碰存量",改已有才算。**DB 加字段不豁免**(仍中型)。
3. **行为变化升级**:若改动虽小但会改变业务判断、权限判断、金额/数量计算、状态流转、筛选条件或对外可观察行为,不得按微型处理;至少升级为小型,并补充对应验收标准与验证项。

**术语界定(消除歧义):**

- **碰存量** = 改"已有"的公共类/接口/方法;纯加新的不算。(→ 给 Service 接口**加方法**是"新开链路"=小型;**改已有方法**才是碰存量=中型)
- **单条调用链** = 一个入口到数据/状态层的完整链路;纵向贯穿多层(Controller→Service→Mapper)仍算**一条**,横向多个独立入口才算多条。
- **完整功能模块** = 围绕一个新实体/新能力,通常含多条链路或需新建表/实体。
- **DB 结构变更** = 加表/加字段/改类型/迁移;仅查改现有表不算。

**定级后查表走流程:**

| 级别 | 产物文件 | Agent |
|---|---|---|
| 微型 | `04` | 不调 |
| 小型 | `00,03,04` | 编码后建议代码评审 |
| 中型 | `00-04` | 设计建议架构评审;编码建议代码+安全评审 |
| 大型 | `00-04` | 关键阶段必须评审 |

> 产物文件编号对应「阶段编号说明」;各级别如何断点续传见下方「进入流程:先定位状态」。

## Bug 修复特殊分支

若用户请求是 Bug 修复,先判断是否为**明显 Bug**。

**明显 Bug** 指满足以下全部条件的问题:

1. 症状和原因高度明确;
2. 修复位置基本确定;
3. 不涉及状态流转、并发、缓存、数据库一致性、外部 API 或复杂业务规则;
4. 不改变既有设计意图、业务规则或对外契约,只是让实现回到既定/期望行为;
5. 能通过最小验证项确认结果。

明显 Bug 可按改动大小分级继续推进,通常为微型或小型。对仅恢复既定/期望行为、且不改变业务规则或对外契约的明显 Bug,修改已有实现不因“碰存量”自动升级为中型。

若修复会重新定义业务判断、权限判断、金额/数量计算、状态流转、筛选条件、对外可观察行为或对外契约,仍按改动大小分级向上取整,至少小型,必要时中型或大型。

若出现以下任一情况,不得按微型直接修,必须进入诊断流程,读取 `references/Bug诊断指南.md`:

1. 无法稳定复现;
2. 用户只描述“偶现、变慢、异常、报错、数据不对”,但没有明确根因;
3. 涉及状态流转、并发、缓存、异步、数据库、外部 API;
4. 修复可能重新定义既有业务规则、权限判断、金额/数量计算、筛选条件或对外契约;
5. 需要猜测原因;
6. 无法通过新增或执行最小验证项覆盖用户描述的具体症状。

诊断流程完成后,再根据实际修复范围重新判定改动级别。

## 进入流程:先定位状态

1. 识别本次改动级别和功能名(用作 `docs/features/<功能名>/` 目录名);若无法判断或存在命名歧义,再向用户确认。
2. 每个功能的产物文件固定命名为:
   - `00-基础资料.md`
   - `01-需求文档.md`
   - `02-设计文档.md`
   - `03-实施清单.md`
   - `04-验证报告.md`
3. 检查 `PROJECT-CONTEXT.md` 是否存在且内容有效:
   - 不存在、内容为空或仅含占位符 → 先进入阶段①,全量扫描
   - 已存在且内容有效 → 阶段①:Project Context 增量确认
4. 查看 `docs/features/<功能名>/` 下已有产出,结合改动级别判断从哪一阶段继续断点。
5. **下一批意图检查**:若目标目录已有 `04-验证报告.md`,但用户意图是“继续下一批”“新增范围”“扩展验收项”或“在上次基础上再做一批”,不得直接按已完成处理。应重新识别本批改动级别和功能名,优先使用新的批次目录(如 `docs/features/<功能名>-<本批目标>/`)承载本批 `00-04` 文档;除非用户明确要求修订同一批,否则不要覆盖已完成批次。
6. **验证报告完成状态检查**:若 `04-验证报告.md` 已存在,不得仅凭文件存在判定完成。必须按 `references/验证报告指南.md` 检查:第 3 节验收结论是否全部通过,第 9 节测试失败处理规则是否无阻塞,第 10 节结论是否明确达到验收或可进入下一批。若仍有失败、无法验证项、未闭环遗留或结论未完成,继续阶段⑥,不得进入阶段⑦。
7. **写回项断点检查**:只有 `04-验证报告.md` 已满足完成状态后,才检查其第 12 节"待写回 Project Context"表格是否有未写回项。只有 `建议动作=写回` 且 `是否已写回=否` 的行参与写回 gating;`建议动作=不写回/暂缓` 的行不阻塞阶段⑦。若存在待写回项,优先处理写回或提醒用户,避免因中断导致稳定事实遗漏。
8. **级别一致性**:若已产出文档表明的级别高于本次判定(有 `01`/`02` 即至少中型),按**更高级别**走、不降级——避免本次改得小、流程却漏掉已有需求/设计。

### 微型改动的状态判断

微型改动只产出 `04-验证报告.md`,不产出 `00-03`。

| 当前状态 | 动作 |
|---|---|
| `PROJECT-CONTEXT.md` 不存在 | 阶段①:读 `references/项目上下文指南.md`,全量扫描 |
| 有上下文,无 `04-验证报告.md` | 阶段⑥:读 `references/AI代码生成约束.md`、本次触及栈的 `references/stack/<栈>.md` 和 `references/验证报告指南.md`,直接实施并产出 `04-验证报告.md` |
| 有 `04-验证报告.md` 但未满足完成状态 | 阶段⑥:继续执行与验证,按 `references/验证报告指南.md` 补齐失败处理、无法验证项和结论 |
| 有 `04-验证报告.md` 且已满足完成状态 | 阶段⑦:写回 `PROJECT-CONTEXT.md`,汇报完成 |

## TDD 节奏

本 skill 默认采用**实用 TDD**。实施和验证阶段必须先读 `references/AI代码生成约束.md`,并按其中的 TDD 规则执行。

简要原则:

- 微型改动:在 `04-验证报告.md` 记录最小验证项。
- 小型改动:测试来源为 `03-实施清单.md` 的 RED 部分。
- 中型/大型改动:测试来源为 `02-设计文档.md` 的测试策略章节。
- 验证阶段只记录执行结果,不临时补造测试用例。

## Agent 调用边界

Agent 调用为**推荐制**,但评审责任不可跳过。本 skill 将专业能力抽象为**角色**;具体角色、调用场景、调用强度、调用输入和降级 checklist 统一见 `references/Agent调用与降级指南.md`。

简要原则:

- 小型改动:通常不强制调用 Agent,但涉及安全、公共接口、数据库或复杂测试时需评估。
- 中型改动:设计阶段建议架构评审/TDD 指导,编码后建议代码评审;涉及安全时做安全评审。
- 大型改动:关键阶段必须评审;若 Agent 不可用,必须按 `references/Agent调用与降级指南.md` 中对应角色的 checklist 自检并记录结论。

## 信任模式

用户可在启动 `/feature-spec` 时明确指定:严格(strict)、普通(normal)、信任(trust)，比如：

- /feature-spec 严格模式 帮我做订单导出
- /feature-spec trust 模式 继续上次任务

默认按**普通模式**推进,高风险必须停下来问用户,低风险自动推进;实施阶段的红线和禁止行为见 `references/AI代码生成约束.md`:

| 模式 | 阶段确认 | 高风险 | 低风险 | 适用场景 |
|---|---|---|---|---|
| `strict` | 每阶段必须等确认 | 必须问 | 必须问 | 新项目、高风险、多人协作 |
| `normal`(默认) | 每阶段汇报关键结论、产物路径和风险;无高风险/阻塞项则自动继续 | 必须暂停提问 | 自动推进 | 日常开发 |
| `trust` | 最后统一汇总 | 真正危险才问 | 自动推进 | 非常熟悉、速度优先 |

> **例外:基础资料的业务意图部分在所有模式下都必须经过用户确认后才推进。** 后续阶段才按上述信任模式自动推进。

## 回退

用户指出某阶段有误,或执行中发现上游依据错误时,不在下游文档偷偷改,回到出错阶段重做:

- 技术栈、构建命令、测试命令或项目工程约定识别错误 → 回 `Project Context`
- 当前项目文件与 `PROJECT-CONTEXT.md` 冲突 → 以当前项目文件为准,回 `Project Context`,增量更新或完整扫描后再继续;若冲突涉及高风险决策,暂停询问用户
- 项目现状理解错 → 回 `Project Context`
- 需求理解错 → 回 `需求文档`
- 设计不合理 → 回 `设计文档`
- 实施拆解错、设计没问题 → 回 `实施清单`
- 原始业务目标错 → 回 `基础资料` 与 `需求文档`
- 实施中用户新增需求或改变原始业务目标 → 回 `基础资料` 与 `需求文档`

实施过程中发现的小修小补(不改变业务目标、架构方向、技术栈判断和项目工程约定),允许在实施清单中记录并同步更新设计文档,不必完全回退。
