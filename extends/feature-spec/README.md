# feature-spec · 功能开发文档引导 skill

一套让 AI 稳定产出"需求 → 设计 → 实施 → 验证"文档、并复用项目既有能力的引导框架。

本 skill **语言/技术栈无关**,支持 **实用 TDD**、**Agent 协作**、**信任分级**、**改动大小分级**和**栈适配包**(Java / Vue+TypeScript / Python 等,见 `references/stack/`)。

---

## 它解决什么

- **不用每次手动让 AI 读一堆文档**:`SKILL.md` 负责判断阶段和断点,只在需要时读取对应的 `references/*.md`。
- **不会每次把所有文档读一遍**:`SKILL.md` 是路由器,每阶段只读对应一份指南(progressive disclosure)。
- **断点续传**:已产出的文档就是流程状态,中途断了能从断点接着干。
- **避免一刀切**:小改动走轻量流程,大改动走完整流程。
- **测试先行**:把 TDD 嵌入设计、实施、验证各阶段。
- **质量兜底**:在关键节点调用架构评审 / TDD 指导 / 代码评审 / 安全评审等专业角色。

---

## 两类文档

| 类型 | 内容 | 位置 |
|---|---|---|
| 持续维护(活文档) | `PROJECT-CONTEXT.md`(稳定信息沉淀 + 易变信息索引) | 每个项目根目录 |
| 按需生成的功能级过程文档 | 00-基础资料 / 01-需求文档 / 02-设计文档 / 03-实施清单 / 04-验证报告 | `docs/features/<功能名>/` |

---

## 产物命名规则

每个功能对应的文档统一放在 `docs/features/<功能名>/` 目录下,文件名固定为:

```
docs/features/<功能名>/
├── 00-基础资料.md
├── 01-需求文档.md
├── 02-设计文档.md
├── 03-实施清单.md
└── 04-验证报告.md
```

阶段文档生成、覆盖、断点判断和写回规则以 `SKILL.md` 及对应 `references/*.md` 为准。

## 使用方式

1. 在项目里对 AI 说"加个 XX 功能" / "改一下 XX" / "重构 XX",或手动运行 `/feature-spec`。
2. 可选:在启动 `/feature-spec` 时说明推进模式;完整模式规则以 `SKILL.md` 为准。
3. AI 会先判断改动大小和功能名,再按对应级别流程推进。
4. 稳定事实的记录、筛选和写回规则见 `references/验证报告指南.md` 第 12 节。

---

## 改动大小分级

feature-spec 会按改动大小动态选择流程。完整分级规则、默认流程和 Agent 调用强度以 `SKILL.md` 为准。

---

## 流程概览

feature-spec 的主流程是:

```text
Project Context → 基础资料 → 需求文档 → 设计文档 → 实施清单 → 执行 + 验证报告 → 写回 Project Context
```

微型/小型改动会自动跳过不必要阶段。完整阶段流转和断点续传规则以 `SKILL.md` 为准。

---

## TDD 节奏

feature-spec 采用**实用 TDD**。测试用例来源、`RED → GREEN → REFACTOR` 执行要求和验证记录规则见 `references/AI代码生成约束.md`、`references/实施清单指南.md` 和 `references/验证报告指南.md`。

---

## 信任模式

feature-spec 支持不同推进模式。可用模式、适用场景和完整推进规则以 `SKILL.md` 为准。

---

## Agent 调用

feature-spec 将专业能力抽象为角色,例如计划评审、架构评审、TDD 指导、代码评审和安全评审。

这些角色可以由当前环境中的 Agent 实现;若目标环境没有对应 Agent,则按 `references/Agent调用与降级指南.md` 中对应角色的 checklist 自检。

完整角色表、触发条件、调用强度、调用输入、输出要求和降级规则以 `references/Agent调用与降级指南.md` 为准。

---

## 回退

阶段判断错误时应回到对应阶段处理,不在下游文档里偷偷修补。完整回退规则以 `SKILL.md` 为准。
