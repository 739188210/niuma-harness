# feature-spec · 功能开发文档引导 skill

一套让 AI 稳定产出"需求 → 设计 → 实施 → 验证 → 写回项目上下文"文档、并复用项目既有能力的引导框架。

本 skill **语言/技术栈无关**,支持 **需求清晰度检查**、**实用 TDD**、**垂直切片实施**、**Bug 诊断**、**Agent 协作**、**信任分级**、**改动大小分级**和**栈适配包**(Java / Vue+TypeScript / Python 等,见 `references/stack/`)。

---

## 它解决什么

- **不用每次手动让 AI 读一堆文档**:`SKILL.md` 负责判断阶段和断点,只在需要时读取对应的 `references/*.md`。
- **不会每次把所有文档读一遍**:`SKILL.md` 是路由器,按阶段读取必要指南(progressive disclosure)。
- **断点续传**:已产出的文档就是流程状态,中途断了能从断点接着干;`04-验证报告.md` 必须达到完成状态后才进入写回。
- **避免一刀切**:小改动走轻量流程,大改动走完整流程;下一批功能会重新识别范围,避免覆盖已完成批次。
- **不靠臆测推进**:需求不清时先查证、再按项目证据默认,最后才询问用户。
- **测试先行**:把 TDD 嵌入设计、实施、验证各阶段,实施清单按可验证行为垂直切片。
- **设计先收边界**:设计阶段检查模块边界、接口边界、deep module、seam、复用点和测试入口。
- **质量兜底**:在关键节点调用架构评审 / TDD 指导 / 代码评审 / 安全评审等专业角色。

---

## 两类文档

| 类型 | 内容 | 位置 |
|---|---|---|
| 持续维护(活文档) | `PROJECT-CONTEXT.md`(稳定信息沉淀 + 易变信息索引) | 每个项目根目录 |
| 按需生成的功能级过程文档 | 00-基础资料 / 01-需求文档 / 02-设计文档 / 03-实施清单 / 04-验证报告 | `docs/features/<功能名>/` |

---

## 产物命名规则

每个功能对应的文档统一放在 `docs/features/<功能名>/` 目录下,文件名固定为:

```
docs/features/<功能名>/
├── 00-基础资料.md
├── 01-需求文档.md
├── 02-设计文档.md
├── 03-实施清单.md
└── 04-验证报告.md
```

阶段文档生成、覆盖、断点判断和写回规则以 `SKILL.md` 及对应 `references/*.md` 为准。

## 使用方式

1. 在项目里对 AI 说"加个 XX 功能" / "改一下 XX" / "重构 XX",或手动运行 `/feature-spec`。
2. 可选:在启动 `/feature-spec` 时说明推进模式;完整模式规则以 `SKILL.md` 为准。
3. AI 会先判断改动大小和功能名,再按对应级别流程推进。
4. AI 会先做需求清晰度检查:能从代码、文档、`PROJECT-CONTEXT.md` 或项目惯例确认的问题不打断用户;只有阻塞业务范围、验收标准或交付目标且无法自证时才询问。
5. 稳定事实的记录、筛选和写回规则见 `references/验证报告指南.md` 第 12 节。

---

## 改动大小分级

feature-spec 会按改动大小动态选择流程。完整分级规则、默认流程和 Agent 调用强度以 `SKILL.md` 为准。

---

## Bug 修复分支

Bug 修复会先区分**明显 Bug**和**不明显 Bug**。明显 Bug 继续按改动大小走轻量流程;不明显 Bug 先按 `references/Bug诊断指南.md` 建立红/绿反馈环,诊断结果记录到 `04-验证报告.md` 的 Bug 修复诊断记录。

微型明显 Bug 可以用用户原始请求和 `04-验证报告.md` 的最小验证项作为授权依据;一旦涉及业务规则重定义、公共契约变化、权限/金额/状态/数据一致性风险,必须升级流程或询问用户。

---

## 设计质量

中型/大型改动的设计阶段不仅描述实现方案,还要检查模块边界、接口边界、deep module 倾向、seam、复用点和测试入口。完整规则见 `references/设计文档指南.md`。

---

## 流程概览

feature-spec 的主流程是:

```text
Project Context → 基础资料 → 需求文档 → 设计文档 → 实施清单 → 执行 + 验证报告 → 写回 Project Context
```

微型/小型改动会自动跳过不必要阶段。完整阶段流转和断点续传规则以 `SKILL.md` 为准。

---

## TDD 节奏

feature-spec 采用**实用 TDD**。测试用例来源、`RED → GREEN → REFACTOR` 执行要求和验证记录规则见 `references/AI代码生成约束.md`、`references/实施清单指南.md` 和 `references/验证报告指南.md`。

实施清单默认按**可验证行为切片**组织,每个切片围绕一个 `REQ-XXX` 或 `TC-XXX` 推进,并承接设计阶段识别的稳定边界、seam、复用点和测试入口。

---

## 信任模式

feature-spec 支持不同推进模式。可用模式、适用场景和完整推进规则以 `SKILL.md` 为准。

业务意图确认是硬性确认点;`trust` 模式下后续阶段通常最后统一汇总,但高风险或阻塞项仍会暂停。

---

## Project Context 写回

`PROJECT-CONTEXT.md` 只沉淀项目级稳定事实,例如模块边界、稳定公共入口、复用能力索引、稳定 seam、测试入口惯例、跨功能可复用的 Bug 诊断反馈环。

当前批次文件清单、`REQ-XXX` / `TC-XXX` 明细、临时 mock、临时 harness、单个 Bug 根因、当前环境状态等一次性信息不写回。唯一正式写回候选队列是 `04-验证报告.md` 第 12 节。

---

## Agent 调用

feature-spec 将专业能力抽象为角色,例如计划评审、架构评审、TDD 指导、代码评审和安全评审。

这些角色可以由当前环境中的 Agent 实现;若目标环境没有对应 Agent,则按 `references/Agent调用与降级指南.md` 中对应角色的 checklist 自检。

完整角色表、触发条件、调用强度、调用输入、输出要求和降级规则以 `references/Agent调用与降级指南.md` 为准。

---

## 回退

阶段判断错误时应回到对应阶段处理,不在下游文档里偷偷修补。完整回退规则以 `SKILL.md` 为准。

若已有 `04-验证报告.md` 但仍有失败、无法验证项、未闭环遗留或结论未完成,继续执行与验证,不进入写回。若用户明确要继续下一批,优先使用新的批次目录,不覆盖已完成批次。
