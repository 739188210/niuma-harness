# dev-workflow

通过七个带门禁的阶段，将需求交付为可上线代码的流程控制型工作流。仅阶段一需要用户参与，后续阶段由 Agent 自驱完成。

## 触发词

当用户要求以下任务时使用本技能：

- implement a feature
- develop a feature
- write code
- 开发功能
- 实现需求
- 写代码

## 目标

通过清晰定义的阶段推进需求落地，每个阶段都有明确的输入、输出、审核门禁、回退规则，并在 `PRD.md` / `DESIGN.md` / `TEST.md` 中跟踪执行状态。

## 前提假设

- 已提供需求来源（工单、issue、简短描述或口头说明）。
- 代码库可访问且可构建。
- 可发现或已存在测试/检查命令。
- 领域相关约定由 Agent 上下文从相关 `CLAUDE.md` / `AGENTS.md` / `CONVENTIONS.md` 自动加载。
- 所有产出必须严格遵循项目规范。

## 阶段总览

| 阶段 | 名称         | 输入                    | 输出                                  | 门禁                      | 失败回退               |
| ---- | ------------ | ----------------------- | ------------------------------------- | ------------------------- | ---------------------- |
| 一   | 产品需求文档 | 用户需求                | `PRD.md` / `PRD.html`                 | 用户确认                  | 回到阶段一             |
| 二   | 产品设计文档 | PRD                     | `DESIGN.md`                           | 自审清单                  | 回到阶段二             |
| 三   | 技术设计     | PRD + DESIGN            | `{name}-api` + `{name}-spec` + Flyway | 自审清单                  | 回到阶段三             |
| 四   | 测试设计     | PRD + DESIGN + 技术设计 | `TEST.md` + 测试数据                  | 自审清单                  | 回到阶段四             |
| 五   | 功能实现     | 全部上游产出            | 代码                                  | 自审清单                  | 回到阶段五、三或二     |
| 六   | 测试         | 代码                    | 单元 + 集成结果                       | 自审清单                  | 回到阶段六、五、四或二 |
| 七   | 页面功能演示 | 完整功能                | 演示记录                              | 内部演示 + 用户确认        | 回到阶段七或五         |

## 自审规则

阶段二至阶段六各自有 phase 特定的自审清单，Agent 自上而下走一遍：

- 任一项未勾选即视为失败。修复产出后只需重新核对受影响的项，无需从清单第一项重启。
- 所有项勾选完成后方可进入下一阶段。
- 阶段一是唯一需要用户显式确认的阶段。
- 阶段七仅在内部演示全部通过后才向用户展示。

各阶段具体的清单与跟踪规则见对应 phase 文件。

## 执行跟踪

`Execution Tracking` 表只维护在 **`PRD.md`** 中。每个阶段完成后更新对应行。`DESIGN.md` 和 `TEST.md` 通过链接引用，不再重复维护表格。

## 升级规则（Escalation Rule）

如果同一阶段连续两次自审循环同一根因仍未取得进展，Agent 必须停止局部修复，回退到可能解决根因的最早阶段。

例如：

- 阶段五实现反复失败，原因是 API 与代码不匹配 → 回退到阶段三技术设计。
- 阶段六集成测试反复失败，原因是测试数据遗漏场景 → 回退到阶段四测试设计。
- 阶段五实现反复失败，原因是 DESIGN 遗漏场景 → 回退到阶段二产品设计。

只有当回退到阶段一之前都无法解决时，才允许向用户求助。

## 跨阶段检查项

下列检查适用于每个阶段。各 phase 的清单只列 phase 特有项；跨阶段检查在每次自审前隐式核对：

1. **正确性** — 产出是否满足输入需求。
2. **完整性** — 是否遗漏场景、字段或分支。
3. **一致性** — 是否遵循项目约定和上游文档；违反视为阻塞项。
4. **模块边界** — 是否尊重领域归属，是否避免与不相关模块耦合，是否只依赖已发布的 API。
5. **清晰性** — 他人是否能独立理解。
6. **安全性** — 是否泄漏 secret、凭据或内部 endpoint。

## 退出条件

- 阶段一产出经用户确认。
- 阶段二至阶段六各自完成自审清单。
- 阶段七通过内部演示并经用户确认。
- `PRD.md` Execution Tracking 表格已更新完整。
- 实现过程中任何偏离 `DESIGN.md` 或 `TEST.md` 的决策，已在 `TEST.md` 的 Notes 中说明，或以注释形式引用相关阶段。
- 模块 `README.md`（如存在）已反映最终实现。
- 所有测试通过。
- 文档与代码一致。
- 未提交任何 secrets、tokens 或密码。
- 功能已转交 `git-workflow` 进行分支、提交、PR 和合并。
