# Web Design Quality Rules

## Purpose

These rules define web design-quality preferences. They help avoid generic template-looking UI without forcing a specific visual style.

## Product fit

Choose visual decisions that fit the product, audience, and task.

A conventional layout is acceptable when it serves the product. Do not add decorative complexity only to avoid looking simple.

Preserve intentional existing product style unless the task asks for a new direction.

## Avoid generic output

Avoid UI that looks like an unmodified template or component-library default.

Watch for equal-weight card grids, generic hero sections, default library styling, dashboard-by-numbers layouts, uniform spacing everywhere, and decorative effects that do not support the content.

Use existing design-system components when available, but adapt composition, spacing, hierarchy, and states to the product context.

## Hierarchy and states

Important content should be visually prioritized through scale, weight, spacing, grouping, alignment, or placement.

Avoid making every section, card, button, or label feel equally important.

Meaningful interactive elements should have designed states for hover, focus, active, selected, disabled, loading, empty, and error when those states apply.

## Responsive composition

Responsive design should preserve hierarchy, not just shrink the desktop layout.

Keep primary actions, key content, and navigation easy to find across breakpoints.

## Data surfaces

For dashboards, charts, and data-heavy UI, design around the user's decision rather than filling cards.

Prefer clear labels, units, comparison, trend, and useful empty/loading/error states. Avoid charts when a table, number, or sentence communicates the answer better.

## Visual restraint

Use motion, depth, texture, illustration, and unusual layouts only when they clarify the experience or support the product tone.

A simple interface with clear hierarchy and states is better than a decorative interface with unclear purpose.
