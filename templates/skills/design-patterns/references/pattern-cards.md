# 设计模式卡

先寻找当前代码中已经存在的变化轴、边界或复杂度，再使用本页。模式名称不是结论；若证据不足，选择更简单的方案。

## 选型速查

| 已观察到的问题 | 优先考虑 | 先确认 | 证据不足时 |
|---|---|---|---|
| 单一、短小且稳定的分支 | 保持条件语句 | 分支是否会扩散到多个调用方 | 保持简单 |
| 两个真实实现，或已承诺在同一近期交付中落地的第二实现 | Strategy | 是否能共享稳定输入/输出契约 | 局部 `if/switch` 或纯函数映射 |
| 外部系统细节泄漏到业务调用方 | Adapter | 是否有模型、错误语义或替换/测试隔离需求 | 在调用点做一次局部转换 |
| 状态、合法/非法迁移或状态副作用复杂 | State / 显式状态机 | 简单条件分支是否仍局部且可测 | 枚举加局部迁移函数或表驱动规则 |
| 创建过程含分支、资源或不变量 | Factory | 创建逻辑是否本身已成为复杂度来源 | 直接或命名构造 |
| 同一进程内多个独立订阅者需要当前调用链通知 | 同步 Observer | 返回、失败、顺序和事务语义是否仍需同步可见 | 显式函数调用或应用服务编排 |
| 事实需跨边界或延后投递给独立消费者 | 异步发布订阅 | 是否能承担投递、重试、幂等、顺序和观测成本 | 显式同步编排 |
| 多处重复组合低层步骤 | Facade / 模块边界 | 组合是否有稳定业务语义 | 局部 helper 或私有方法 |
| 只有“未来可能扩展” | 不引入模式 | 是否存在已确认的近期变化 | 保持简单并在变化出现时复查 |

## Strategy（策略）

### 解决的问题

同一业务目标存在两个及以上稳定、可替换的规则、算法或渠道实现，调用方不应随着实现增加而持续扩展条件分支。

### 采用证据

- 当前已有两个真实实现；或第二个实现有明确需求、范围和近期交付承诺，并会在同一变更集中落地；
- 选择逻辑已在多个调用点重复，或新增实现会修改核心流程的多个分支；
- 各实现能够共享清晰、稳定且有意义的输入和输出契约；
- 调用方只关心结果，不应知道具体实现细节。

### 不适用条件

- 只有一个实现，且第二实现没有明确范围与交付承诺；
- 分支局部、短小且稳定；
- 各实现不能共享有意义的契约，只能靠宽泛参数或大量类型判断凑在一起；
- 唯一依据是“以后可能增加”。

### 简单替代

局部 `if/switch`、一个私有选择函数，或数据驱动的纯函数映射。

### 最小实现边界

第二实现已实际落地后，保留一个稳定策略契约、两个真实实现和一个明确选择位置。调用方依赖契约，不需要注册中心、插件框架、空策略或占位实现。

### 验证重点

- 同一输入在每个策略下的行为测试；
- 选择逻辑按条件选择正确实现；
- 调用方不依赖具体策略类型或实现细节。

### 常见过度设计信号

为一个策略建立接口、空默认实现、自动扫描注册或多余工厂；第二实现尚未落地就预建策略层。

## Adapter（适配器）

### 解决的问题

隔离外部 API、SDK 或遗留模块的类型、字段、协议和错误语义，防止其向业务调用方扩散。

### 采用证据

- 多个调用方直接处理第三方类型、原始响应或协议细节；
- 外部错误模型正在迫使业务层写供应商特定判断；
- 外部依赖需要替换、升级、隔离测试或兼容遗留接口；
- 转换边界具有稳定业务含义，而非一次性的字段重命名。

### 不适用条件

- 只有一处简单、稳定的外部调用；
- 转换只包含一次局部字段映射，且没有泄漏或复用需求；
- 新增包装层不能隔离类型、异常或传输细节。

### 简单替代

在调用点完成一次局部转换，并把转换保留在当前模块私有范围。

### 最小实现边界

适配器负责调用外部依赖和映射数据、异常；业务调用方只依赖本项目定义的输入、输出和错误语义。不要把外部 SDK 的类型穿过适配器边界。

### 验证重点

- 正常数据和边界数据的映射测试；
- 外部异常到项目错误语义的映射测试；
- 业务层不直接依赖外部类型或协议字段。

### 常见过度设计信号

只包一层原样转发；一个适配器暴露全部 SDK；为不存在的第二个供应商预建多级抽象。

## State / 显式状态机

### 解决的问题

实体具有明确状态、合法与非法迁移、状态相关副作用或并发约束，现有条件分支已难以推理、覆盖或防止非法路径。

### 采用证据

- 状态集合、允许迁移和禁止迁移可以明确列举；
- 同一迁移规则或非法迁移判断在多个调用点重复；
- 状态变化会触发明确副作用，且副作用归属、事务边界或重复请求处理需要统一；
- 现有 `if/switch` 已分散、嵌套或难以覆盖合法与非法路径。

### 不适用条件

- 状态少、迁移少，条件判断局部且容易测试；
- 只有状态字段，没有状态专属行为或迁移规则；
- 无法定义合法状态集合或迁移语义，只是在替换杂乱业务规则。

### 简单替代

状态枚举或受控字段，加集中但局部的迁移函数；迁移规则有限时可用迁移表。不要为每个枚举值机械创建状态对象。

### 最小实现边界

先选择最小表达：状态枚举加迁移校验适合有限规则；迁移表适合规则需集中审查；只有状态专属行为明显且持续扩张时才考虑状态对象。非法迁移必须显式拒绝。副作用要明确发生在迁移前、迁移后或事务提交后；重复请求、并发版本冲突和幂等边界不得被状态机隐藏。

### 验证重点

- 每条合法迁移得到预期新状态；
- 每条非法迁移被拒绝且不产生副作用；
- 副作用、事务提交、重复请求和并发冲突符合承诺；
- 迁移表或状态对象的规则可被独立覆盖。

### 常见过度设计信号

为两三个稳定条件建立状态对象层级；用状态机掩盖未定义的业务规则；未定义非法迁移和副作用边界。

## Factory（工厂）

### 解决的问题

对象创建过程本身已有真实分支、复杂默认值、资源初始化、不变量校验或具体实现选择，不能再被安全地视为简单构造。

### 采用证据

- 创建根据输入、配置或环境选择不同具体实现；
- 创建过程负责重要默认值、校验或资源生命周期；
- 多个调用方重复相同的构造和校验步骤；
- 直接构造使调用方必须知道不应暴露的初始化细节。

### 不适用条件

- 只有一条直接构造路径；
- 工厂只是把 `new` 包一层；
- 创建逻辑只有一个调用方且提取后没有更清晰的语义。

### 简单替代

直接构造函数、命名构造函数，或当前模块内私有的创建 helper。

### 最小实现边界

工厂仅封装创建决策、必要校验和初始化，不接管后续业务行为。无须创建“抽象工厂”或产品族，除非当前确有多个相关产品族。

### 验证重点

- 每种有效输入得到正确对象和默认值；
- 非法类型、缺失参数和不变量违反能清晰失败；
- 调用方不再重复关键初始化规则。

### 常见过度设计信号

单实现工厂、无产品族的抽象工厂、把业务流程塞进工厂，或用工厂掩盖依赖注入问题。

## 同步 Observer

### 解决的问题

同一进程内的多个独立订阅者需要在当前调用链中响应一个事实，生产者不应直接编排所有订阅者，但同步完成、失败传播、顺序或事务语义仍必须可见。

### 采用证据

- 当前已有两个及以上独立订阅者；
- 订阅者可独立演进，生产者不应依赖其实现；
- 订阅者在当前调用链中完成是正确语义；
- 能明确失败传播、顺序和事务边界。

### 不适用条件

- 只有一个消费者；
- 生产者实际需要明确编排各步骤或依赖其返回值；
- 需要跨进程、延后投递、重试或最终一致性；这些属于异步发布订阅问题。

### 简单替代

显式函数调用，或由应用服务在清晰事务/错误边界内编排多个步骤。

### 最小实现边界

观察者在当前调用链中同步执行。明确订阅顺序是否承诺、订阅者失败是否阻断主流程，以及事务是否涵盖通知；不要用同步观察者隐藏关键业务编排。

### 验证重点

- 正确条件下通知预期订阅者；
- 订阅顺序、失败传播和同步完成语义符合承诺；
- 事务边界内外的副作用符合设计。

### 常见过度设计信号

进程内单消费者也引入总线；用观察者绕开明确依赖；需要可靠异步投递却仍使用进程内同步通知。

## 异步发布订阅

### 解决的问题

已发生的事实需要跨进程、跨模块边界或延后投递给独立消费者，调用方接受明确的异步和最终一致性语义。

### 采用证据

- 当前已有两个及以上独立消费者，或有真实的跨边界、削峰、独立演进或时序解耦需求；
- 主流程不依赖消费者在当前调用链中的返回值或即时完成；
- 能明确事件在事务提交前还是提交后发布；
- 能设计投递持久化、失败处理、重试、幂等、顺序、死信/告警和观测策略。

### 不适用条件

- 关键成功条件、返回值、事务原子性或严格顺序仍属于主流程；
- 没有投递可靠性、失败归宿或消费者幂等设计；
- 只是为了绕开正常同步调用，或只有一个消费者。

### 简单替代

显式同步编排；除非异步语义有真实收益，不要以消息机制替代清晰函数调用。

### 最小实现边界

事件表达已发生的事实，而非命令。明确提交后发布、投递存储和消费者边界；消费者必须按承诺处理重复、失败、顺序和可观测性。投递失败不能静默吞掉，也不能把关键主流程成功语义隐藏在异步链中。

### 验证重点

- 事实成立后在正确时点产生和投递事件；
- 消费者幂等、重试、失败归宿和告警符合承诺；
- 若承诺顺序、隔离或最终一致性，测试和观测覆盖对应承诺。

### 常见过度设计信号

用异步消息隐藏关键同步调用；没有幂等和失败策略；假定进程内事件等同可靠消息；为单消费者建立复杂投递系统。

## Facade（外观）/ 模块边界

### 解决的问题

多个调用方反复组合多个低层步骤，而这个组合已经具有稳定的业务语义；上层不应理解低层协作细节。

### 采用证据

- 相同或高度相似的跨模块编排已在多个调用方出现；
- 组合步骤存在稳定名称、错误边界、事务边界或业务含义；
- 上层调用方因了解太多低层模块而难以理解、测试或演进；
- 新入口能减少真实依赖面，而不只是增加转发层。

### 不适用条件

- 只出现一次，且尚未证明会复用；
- 步骤没有稳定业务语义；
- 外观只是转发一个调用，或把所有低层 API 全部重新暴露；
- 当前模块内一个私有 helper 已能清楚表达意图。

### 简单替代

当前模块内的局部 helper、私有方法，或把少量重复逻辑保留在一个现有服务中。

### 最小实现边界

外观只暴露稳定的上层意图，封装必要的低层编排、错误传播和事务边界；不要变成无边界的“万能服务”。

### 验证重点

- 上层通过稳定入口获得正确业务结果；
- 失败、补偿或事务行为与原有语义一致；
- 调用方不再需要依赖或组合不必要的低层模块。

### 常见过度设计信号

仅转发一次的服务层；把不相关能力塞入同一个 Facade；为了目录分层而建立没有业务语义的中间层。
