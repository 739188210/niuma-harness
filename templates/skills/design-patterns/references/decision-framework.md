# 决策框架

先根据当前代码与已确认的近期需求收集证据，再决定是否需要新增抽象。模式名称不是起点。

## 1. 观察证据

先阅读相关实现、调用方、测试和项目既有惯例，记录可定位的事实：

- **变化轴**：规则、渠道、供应商、协议或创建路径是否已有多个真实实现，或第二个实现是否有明确需求、范围和近期交付承诺；
- **边界泄漏**：外部类型、字段、错误语义或传输细节是否进入业务调用方；
- **分支扩散**：同类条件选择是否已在多个调用点重复；
- **重复编排**：多个调用方是否重复组合低层步骤，且组合具有稳定业务语义；
- **状态复杂度**：状态、允许迁移、禁止迁移或状态相关副作用是否难以通过现有条件分支理解和验证；
- **事件语义**：调用方是否需要同步结果、失败传播或同一事务语义；若不需要，是否存在跨边界、独立演进或时序解耦的真实需求；
- **验证困难**：当前结构是否迫使测试依赖不应暴露的外部细节，且局部替身或参数注入不足以解决。

不要将“未来可能扩展”、单一分支、一次性任务或未验证猜测作为证据。

## 2. 比较三档方案

每个结论都必须比较以下方案，并说明每档的当前收益、额外代价和不选原因：

| 方案 | 适用情况 | 典型代价 |
|---|---|---|
| 保持简单 | 单一、局部、稳定的实现 | 后续变化出现时才需要重评估 |
| 局部抽取 | 重复有限，私有函数或模块已能表达边界 | 不能隔离真正的多实现或外部边界 |
| 最小模式方案 | 已存在稳定变化轴、关键边界或复杂状态，前两者无法控制复杂度 | 新增契约、边界和相应验证 |

选择保持简单时仍要写出复查触发条件。选择局部抽取或最小模式时，要说明为什么更低一档不足。

## 3. 选择最小可验证方案

- 简单方案能够满足当前需求时，明确选择它，并写出重新评估的触发条件。
- 需要局部抽取时，先在当前模块内建立清晰私有边界；不要提前升级为公共框架。
- 只有模式能消除已观察到的耦合、边界泄漏或扩展成本时，才读取对应模式卡并采用其最小形态。
- 状态迁移复杂度进入 State / 显式状态机条目；同步通知进入同步 Observer 条目；异步投递进入异步发布订阅条目。三者不能互相替代。
- 每个新增边界都必须有一个真实调用点；不要先建立无使用方的通用框架。

## 4. 定义验证

推荐方案必须说明：

- 职责与依赖方向；
- 哪个真实调用点使用该结构；
- 保持的行为与新增的行为；
- 应覆盖的边界、异常、选择逻辑或状态迁移；
- 用什么测试或可观察结果证明该抽象降低了问题，而不是只增加层级。

## 5. 记录取舍

在任务记录中保留观察证据、三档比较、被拒绝方案、选择原因、复查条件和验证结果。只有跨模块、公共接口、长期依赖或难回滚的决策，才建议另行记录长期 ADR；不要把每次局部实现选择升级为长期架构决策。
